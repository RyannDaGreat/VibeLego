# Plan: Clara Slope Bricks + Anatomy enable_key + Manifest Update

## 1. Anatomy enable_key refactor (quick win, do first)

**Problem**: The anatomy section in `blender_watcher.py` has bespoke show/hide logic:
```python
box.prop(props, "show_anatomy")
if props.show_anatomy:
    box.prop(props, "anatomy_region")
```
This duplicates what `enable_key` already does generically. Should be converted to a section with `enable_key: "show_anatomy"`.

**Approach**: Move anatomy from hardcoded draw logic into a proper section in `panel_common.py`:
- Add `ANATOMY_SECTION` dict with `enable_key: "show_anatomy"` and params for `show_anatomy` (bool) and `anatomy_region` (enum)
- Import it in each `panel_def.py` and add to SECTIONS
- Remove the hardcoded anatomy block from `blender_watcher.py` draw_panel
- Remove the hardcoded anatomy property registration from `_build_panel_classes`

**Complication**: The anatomy props have an `update=_on_anatomy_toggle` callback, and the enum items come from `_anatomy_region_items` which is set at registration time. The generic `_make_bpy_property` needs to handle `update` callbacks and dynamic enum items. Currently it doesn't.

**Two sub-options**:

**(A) Full generalization**: Add `update` and `items` support to `_make_bpy_property`, move anatomy into SECTIONS. Clean but more work — needs `_make_bpy_property` to accept a callback reference, which means the section data structure needs to carry function references (currently it's pure data).

**(B) Lightweight**: Keep anatomy as a special section in `blender_watcher.py` but use the `enable_key` draw pattern. The section dict is built at registration time (when `_on_anatomy_toggle` and `_anatomy_region_items` are available), not in `panel_def.py`. Anatomy section is appended to `_sections` after SECTIONS are loaded.

**Recommendation**: **(B)** — keeps the panel_def files as pure data (no function references), and anatomy is display-only (not a build param), so it doesn't belong in SECTIONS alongside parametric params anyway. The change is just in `blender_watcher.py`: build a section dict for anatomy with `enable_key`, append it to `_sections`, and delete the hardcoded draw block.

### Files changed:
- `blender_watcher.py`: Build anatomy section dict at registration, append to `_sections`, remove hardcoded draw block and hardcoded property registration

### Geometry: `clara_slope()` in `clara_lib.py`

Based on the LEGO slope pattern but adapted for Clara lattice. Key differences from `lego_slope()`:

1. **Shell**: LEGO slope uses `Box` → `split`. Clara needs to handle corner_radius + wall_taper. Two cases:
   - **No taper, no corner radius**: Same as LEGO — `Box` → `split`
   - **With corner_radius and/or taper**: Build the shell via loft (same as `clara_brick` taper path), then `split` the result with the slope plane. The loft produces the tapered shell, the split cuts the slope.

2. **Cavity**: Same approach as LEGO slope — `Box(inner_x, inner_y, cavity_z)` → `split` with offset cavity cut plane. Cavity stays sharp (no corner radius on cavity, same as `clara_brick`).

3. **Lattice**: Build lattice struts as a separate Part (2D sketch → extrude to `cavity_z`), then clip with `& sloped_cavity`. Same pattern as LEGO tubes but with diagonal struts instead of cylinders.
   - The lattice is currently built inside the main `BuildPart` in `clara_brick()`. For the slope, it needs to be built in a separate `BuildPart` (like LEGO slope tubes) so it can be boolean-intersected with `sloped_cavity`.

4. **Studs**: Only on flat rows, same approach as LEGO slope. Compute `flat_xy` positions, place studs (with optional stud taper via `_build_stud`).

5. **Fillet**: Same `fillet_above_z` with `z_threshold=cavity_z`. Known issue from concerns: LEGO slope 2x4 fillet fails — Clara might too. Handle gracefully.

6. **Text**: Same as LEGO slope — only on flat stud positions.

### Slope plane math (same as LEGO, proven correct)

```
hinge_y = -outer_y/2 + flat_rows * PITCH
slope_dy = outer_y/2 - hinge_y
slope_dz = height - WALL_THICKNESS
normal = (0, slope_dz, slope_dy)

cut_plane = Plane(origin=(0, hinge_y, height), x_dir=(1,0,0), z_dir=normal)

# Cavity cut plane offset inward by FLOOR_THICKNESS
normal_mag = sqrt(slope_dz^2 + slope_dy^2)
cavity_cut_plane = Plane(
    origin=(0,
            hinge_y - FLOOR_THICKNESS * slope_dz / normal_mag,
            height - FLOOR_THICKNESS * slope_dy / normal_mag),
    x_dir=(1,0,0), z_dir=normal)
```

### Refactor consideration: shared slope math?

Both `lego_slope` and `clara_slope` compute the same slope planes. Could extract `_slope_planes(outer_y, height, flat_rows)` into `common.py`. But:
- It's ~8 lines of math, used in 2 places
- The LEGO slope code is stable and battle-tested
- Extracting creates a coupling between the two systems that doesn't exist currently

**Decision**: Don't extract. Copy the slope plane math. It's short and battle-tested. Less risk than refactoring two working functions.

### Shell construction for slope + taper + corner_radius

This is the trickiest part. Three approaches:

**(A) Loft full shell → split**: Build the full tapered/rounded shell via loft (identical to `clara_brick`), then `split` with slope plane. The split cuts through the loft cleanly.
- Pro: Reuses existing loft code exactly
- Pro: Proven `split` approach from LEGO slope
- Con: `split()` on loft geometry — untested, could produce bad topology (concerns.md warns about split on complex geometry)
- Con: Taper is at the top of walls, but slope cuts through the top — the taper zone may get entirely removed on the sloped portion, wasting computation

**(B) Build outer box → split → then apply taper only to flat portion**: Build a plain box, split off the slope, then apply taper only where the flat deck remains.
- Pro: Simpler geometry for split
- Con: Taper on a split surface is geometrically complex — the taper profiles need to be clipped to the slope line

**(C) Start simple — no taper/corner_radius for slopes**: v1 of `clara_slope` ignores taper and corner_radius. Just Box → split → cavity → lattice → studs. Add taper+radius for slopes later if needed.
- Pro: Fastest to implement, mirrors LEGO slope exactly
- Pro: The user might not even want taper on slopes (taper is for print tolerance on vertical walls; slope face doesn't need it)
- Con: Corner radius on slopes would look nice

**Recommendation**: **(C)** for v1. The slope is already a complex feature. Taper+radius on slopes can be added later. The params still exist in the panel (they just won't affect the slope geometry if slope is enabled), or we could note this in the description.

Actually, **(A)** is also reasonable and not that hard. The loft produces a solid, and `split` on solids is well-behaved (the concerns.md warnings are about splitting *hollow* geometry like tubes). The tapered shell is a solid — it should split cleanly.

**Final decision**: Try **(A)** first. If split on loft geometry fails, fall back to **(C)**.

### parametric.py changes

`_build()` checks `enable_slope`:

if params.get("enable_slope", False):
    flat_rows = int(params.get("slope_flat_rows", 1))
    return clara_lib.clara_slope(studs_x, studs_y, flat_rows=flat_rows, ...)

### Anatomy classification for slopes

`classify_face` in `panel_common.py` currently doesn't know about slopes. The slope face (angled surface) would be classified as "walls" or "default" based on Z position. This is probably fine for v1 — a "slope" anatomy region can be added later if needed.

### Known risks from concerns.md

1. **split() on complex geometry**: Warned about for hollow cylinders (tubes). Our shell is solid, should be OK. The lattice is built separately and clipped with `&`, same safe pattern as LEGO tubes.
2. **Cavity offset plane**: MUST offset by FLOOR_THICKNESS along slope normal. Without this, zero material between slope face and cavity (proven bug from LEGO slope).
3. **Fillet on slopes**: LEGO slope 2x4 fillet already known to fail. Expect same for Clara. May need to skip fillet for slopes or use a smaller radius.

## 4. Internals anatomy sub-regions

Split "internals" into sub-regions, same overlapping parent/child pattern as walls_all and studs_all:

| Region | Color | Classification |
|--------|-------|----------------|
| Internals (all) | — | Parent group: internal_walls + internal_ceiling |
| Internal Walls | Purple (existing internals color) | Vertical faces inside cavity bounds (lattice struts, tubes) |
| Internal Ceiling | Lighter purple / lavender | Horizontal downward-facing faces at Z ≈ cavity_z, inside inner bounds |

**Classification logic** in `classify_face`:
- Currently: `inside_inner and cz > tol and cz < cavity_z + tol` → "internals"
- Split: check normal direction:
  - `nz < -0.5` and `abs(cz - cavity_z) < tol` and inside inner → "internal_ceiling" (deck underside)
  - Everything else inside inner cavity → "internal_walls" (struts, tubes, etc.)

Note: the existing "Deck underside / ceiling" classification (line 260-261 in panel_common.py) already catches `abs(cz - cavity_z) < tol and nz < -0.5 and inside_inner` and classifies it as "deck". We need to decide: is the underside of the deck a "deck" region or "internal_ceiling"?

**Decision**: The deck underside is "internal_ceiling" — it faces INTO the cavity, not up. The "deck" region should be only the top surface where studs sit. This is a reclassification: move the existing deck-underside check from returning "deck" to returning "internal_ceiling".

**Cavity**: The empty space (void) inside the brick. Already named "Cavity" in the glossary. Not a face region — it's the absence of material. Ensure it's in the manifest glossary.
