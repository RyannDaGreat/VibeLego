build123d · Architecture Report

build123d → Blender Live Preview:
Architecture & Alternatives

March 2026 Platform: macOS ARM64 · Blender 5.0.1 Status: Implementation in progress
§1

Executive Summary

▶

This document describes the architecture of a zero-button live preview system for build123d CAD models in Blender. The goal is to eliminate the friction in a Claude-assisted modeling workflow: Claude edits a .py source file, and the resulting 3D mesh automatically appears — or updates — in a Blender viewport without any manual intervention. Materials, transforms, and scene configuration set by the user are preserved across every mesh update.

The chosen approach is a subprocess + STL file watcher: Blender runs a Python script that polls the source file's modification time every 0.5 seconds. When it detects a change, it spawns a subprocess using the system Python (which has build123d installed in a venv), the subprocess exports the model to _preview.stl, and Blender then imports the STL and applies the geometry to an existing scene object via mesh.clear_geometry() + mesh.from_pydata(). Seven alternative approaches were investigated and rejected; their tradeoffs are documented here for posterity.

§3

The Chosen Approach: Subprocess + STL File Watcher

▶

Entry point: run.sh

The shell script performs environment detection: it locates the Blender executable (checked at /Applications/Blender.app/Contents/MacOS/Blender) and the active Python venv. It then invokes Blender with three important flags: 

blender --factory-startup \
        --python blender_watcher.py \
        -- source.py
  • --factory-startup: launches with a clean default scene (no user preferences, no saved UI state)
  • --python blender_watcher.py: executes the watcher script immediately on startup
  • -- source.py: passes the model path to the script via sys.argv

Scene setup: blender_watcher.py (startup phase)

On load, the watcher script clears the default cube, camera, and light, then sets up a CAD-appropriate viewport: orthographic projection, studio HDRI lighting (or equivalent), and a neutral grey world background. This runs once. The script then registers a timer callback and returns control to Blender's event loop.

Timer polling (the hot path)

bpy.app.timers.register(check_fn, first_interval=0.5, persistent=True) schedules check_fn every 0.5 seconds. The function reads the source file's os.stat().st_mtime and compares it to the last-seen mtime. If unchanged, it returns 0.5 to reschedule. If changed, it proceeds to the update path.

Why mtime, not content hash?
A known issue with CQ-editor's auto-reload (GitHub issue #280) is that it "randomly stops working." CQ-editor uses QFileSystemWatcher, which relies on OS file event delivery — an unreliable mechanism. Mtime polling is simpler and cannot miss a change as long as the filesystem updates mtime on write (which all relevant filesystems do).

Subprocess: building the STL

When a change is detected, Blender spawns a subprocess running the system Python (the venv that has build123d installed), not Blender's embedded Python 3.11. This is the key isolation boundary: build123d depends on cadquery-ocp, a large C++ binary that cannot be installed into Blender's Python without risking binary compatibility issues. The subprocess simply runs source.py, which is expected to write _preview.stl in the working directory.

STL import and mesh update

Blender imports the STL using the modern operator: 

bpy.ops.wm.stl_import(filepath="_preview.stl")
Blender 5.0 API change
The old operator bpy.ops.import_mesh.stl() was removed in Blender 4.1+. The replacement bpy.ops.wm.stl_import() is a C++ rewrite, 3–10× faster, and ships as a built-in (not an addon). Any version-safe wrapper must detect the available operator at runtime.

After import, the script extracts vertices and faces from the temporary import object, then updates the persistent preview object using: 

preview_mesh.clear_geometry()
preview_mesh.from_pydata(verts, [], faces)

The temporary import object and its datablock are then deleted to prevent orphan data accumulation. On first run, the preview object is created; on subsequent runs, only its geometry is replaced.

Why clear_geometry() + from_pydata() instead of datablock swap?

The obvious alternative — obj.data = new_mesh — does not preserve material slots when the material link mode is 'DATA' (the default in Blender). Material assignments live on the datablock, so swapping the datablock loses them. clear_geometry() clears only the geometry (vertices, edges, faces) while leaving material slots intact. This has been the correct approach since Blender 2.81.

Latency characteristics

The total round-trip time from file save to viewport update is approximately:

  • Mtime poll delay: 0 – 500 ms (average ~250 ms)
  • Subprocess startup + build123d import: ~300 – 800 ms (first run; subsequent runs may be faster if Python caches imports)
  • STL export: ~50 – 200 ms depending on mesh complexity
  • Blender STL import + mesh update: ~50 – 150 ms

Total: ~700 ms – 1.7 s. This is entirely acceptable for a Claude-assisted workflow where the human conversation is the bottleneck, not mesh transfer.

§9

Known Risks

▶

Click each risk to expand the mitigation strategy.

Medium Python version mismatch

If the user upgrades Homebrew Python (e.g., from 3.13 → 3.14), the build123d venv will break. setup.sh must detect the Python version and recreate the venv if needed. The watcher script should fail loudly (non-zero subprocess exit) rather than silently showing a stale or empty mesh.

High Blender API instability

The STL import operator already changed once: bpy.ops.import_mesh.stl() was removed in Blender 4.1 and replaced with bpy.ops.wm.stl_import(). The watcher script must include a version-safe wrapper that detects which operator is available at runtime, rather than hardcoding one version.

Medium Tessellation quality

The default STL export tolerances (linear_tolerance=1e-3, angular_tolerance=0.1) may produce coarse meshes for small or highly-curved geometry, or overly dense meshes for large flat geometry. These should be configurable via command-line arguments passed through run.sh.

Low Large model performance

For complex assemblies (millions of triangles), STL export and reimport can be slow. Not a concern for typical Claude-assisted modeling (simple parametric parts), but worth noting for future use. The socket IPC approaches (Alternatives B, C, D) would address this if it becomes a real constraint.

Low Orphan data accumulation in Blender

Every STL import creates a temporary mesh datablock and object. If not explicitly deleted after copying geometry, Blender accumulates orphan datablocks (visible as zero-user meshes in the outliner). The watcher script must call bpy.data.meshes.remove() and bpy.data.objects.remove() on the temporary import object after extracting its geometry.

Low CQ-editor reliability regression (reference)

CQ-editor's auto-reload "randomly stops working" (GitHub issue #280) because it uses QFileSystemWatcher, which depends on OS file event delivery. This project uses mtime polling instead, which is immune to this failure mode. Documented here as rationale for the polling decision.

Medium Subprocess output swallowed silently

If source.py raises an exception (build123d error, syntax error, etc.), the subprocess exits non-zero. The watcher must capture and surface this output — ideally print it to Blender's console and/or overlay it in the viewport — rather than silently failing to update the mesh. Silent failure here would be confusing and hard to debug.
